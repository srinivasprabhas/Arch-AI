  - Created `types/template.ts` — single source of truth for the template wire format. `StarterTemplate` has `id` / `name` / `description` / `category` / `version` / `source` / optional `preview` / `nodes` / `edges`, where `nodes` and `edges` are the existing `CanvasNode` / `CanvasEdge` types from `types/canvas.ts` so the template snapshot and the live collaborative canvas can never drift apart. `TemplateSource` is a discriminated union (`builtin` | `organization` | `user` | `ai`) so future template providers slot in without breaking call-sites — the registry already exposes them as `StarterTemplate` regardless of source
  - Created `lib/templates/templates.ts` — moved the existing three starter snapshots (Microservices Architecture, CI/CD Pipeline, Event-Driven System) out of `components/editor/starter-templates.ts` and into the new module, adding `category` (`"system-design"` / `"devops"`), `version: 1`, and `source: { kind: "builtin" }`. The helper builders (`makeNode`, `makeEdges`, `COLOR`, `SHAPE_DEFAULTS`) live alongside the data so future authors have one file to grep
  - Created `lib/templates/template-utils.ts` — three pure functions that drive the entire clone pipeline:
    - `cloneTemplate(template, options?)` — deep-clones nodes + edges, generates fresh ids via `crypto.randomUUID` (with a `${Date.now()}-${random}` fallback for older runtimes), and remaps every edge endpoint through a `Map<oldId, newId>`. Edges whose source/target don't resolve are dropped via `flatMap` rather than passed through with a stale id (silent corruption would be worse than a missing edge). Strips transient interaction flags (`selected`, `dragging`) so the persisted snapshot is minimal. Accepts a custom `generateId` to make tests deterministic. New ids are prefixed with the template id so a quick visual scan of the resulting room makes the provenance obvious
    - `generateTemplateProjectName(template, existingNames)` — picks "Microservices Architecture", "Microservices Architecture 2", "Microservices Architecture 3", … honoring whatever the user already has in their workspace. Identical algorithm to `CreateProjectButton`'s `Untitled scene N` logic so the two create paths feel the same
    - `sanitizeTemplateData(cloned)` — strips everything except `id` / `type` / `position` / `style` / `data` on nodes and `id` / `type` / `source` / `target` / `data` on edges. Defensive against React Flow / Liveblocks version drift — the persisted JSON only contains keys the canvas actually reads
  - Created `lib/templates/index.ts` — barrel + central O(1) registry. `getTemplate(id)` returns `StarterTemplate | null`; `listTemplates()` returns the registration-order list (used by the gallery). The registry is a `ReadonlyMap` rather than an object literal so additional sources (org / user / AI) can push into the same lookup without colliding with object-key constraints. Also re-exports the utils so `@/lib/templates` is the single import everywhere
  - Created `app/api/projects/from-template/route.ts` — `POST` handler. Validates Clerk identity (401 on missing), parses body through `RequestSchema = z.object({ templateId: z.string().min(1) })` (400 on invalid shape, 400 on non-JSON body), looks up the template via `getTemplate(parsed.data.templateId)` (404 on unknown). Server-side flow:
    1. `prisma.project.findMany({ where: { ownerId }, select: { name: true } })` to gather existing names
    2. `generateTemplateProjectName(template, existing.map((p) => p.name))` to pick a non-colliding name
    3. `prisma.project.create({ data: { ownerId, name } })` first — so the new project's id can be used as the deterministic blob pathname (`canvas/{projectId}.json`), identical to the autosave PUT path from Feature 21 (one canvas storage layout, not two)
    4. `cloneTemplate(template, { idPrefix: template.id })` → `sanitizeTemplateData(cloned)` → `JSON.stringify({ nodes, edges })`
    5. `put('canvas/${project.id}.json', payload, { access: "private", contentType: "application/json", allowOverwrite: true, addRandomSuffix: false })` — same flags the existing canvas PUT uses, so the GET route reads the seeded snapshot through the exact same code path it reads any other canvas
    6. `prisma.project.update({ where: { id: project.id }, data: { canvasJsonPath: blob.url } })`. On any throw between steps 4–6 the half-created `Project` row is rolled back via `prisma.project.delete` so the user is never left with an empty workspace pointing at no canvas. Rollback failures are logged but don't mask the original error
    7. Returns `{ id, name, templateId, templateVersion }` with `201`

## Architecture Decisions

- shadcn/ui chosen as component library (see ui-context.md) — no custom design
  system, components live in components/ui/.
- Tailwind v4 CSS-variable-first approach: all theme tokens defined as CSS custom
  properties in globals.css and exposed to Tailwind via @theme inline.
- Dark-only theme: no light mode variant, all CSS variables set on :root with
  dark values directly (no .dark class toggling needed).
- tailwindcss-animate added as @plugin in globals.css to support dialog
  animate-in/animate-out utilities.
- @theme inline maps both shadcn semantic names (--background, --primary, etc.)
  and project-specific names (--color-brand, --color-copy-primary, etc.) so both
  sets of utility classes are available.

## Architecture Decisions (continued)

- Next.js 16 renames `middleware.ts` → `proxy.ts` at project root. Must export a default or `proxy` function. All Clerk middleware logic lives there.
- Clerk appearance API (v7 + @clerk/ui) uses `theme` (not `baseTheme`) and `colorForeground` / `colorInput` / `colorInputForeground` — not the deprecated `colorText`/`colorInputBackground` names.
